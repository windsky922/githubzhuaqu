# GitHub 项目研究 Agent v1 审查报告

本文沉淀一次面向完整产品闭环的项目审查。结论来自当前代码、README、API、RAG、订阅推送和近期开发规划，不追求一次性强化项目研究深度，而是先把项目做成可演示、可运行、可降级、可持续迭代的 Agent 产品骨架。

## 一、当前产品定位

GitHub Weekly Agent 的长期定位不是普通周报工具，而是 GitHub 项目研究 Agent。

核心闭环应当是：

```text
采集信号
-> 沉淀证据
-> 检索相关上下文
-> 生成判断
-> 用户反馈
-> 调整推荐
-> 订阅推送
-> 自动运行
```

每周热点周报应继续保留，但它应逐步变成订阅推送体系中的一个稳定频道，而不是整个项目的唯一产品形态。

## 二、已经完成的基础能力

当前项目已经具备完整产品骨架的大部分底座：

1. GitHub Trending 周榜优先采集，GitHub Search 辅助补充。
2. 热点筛选、评分、Star 增量、质量分、风险提示和推荐理由。
3. Kimi 周报生成，失败时规则版降级。
4. Telegram、飞书、企业微信链接推送。
5. GitHub Pages 归档页、项目页、推荐页、订阅页和管理页。
6. SQLite 派生索引，覆盖运行记录、仓库、入选项目、任务、订阅、RAG、反馈和通知事件。
7. FastAPI 后端，覆盖项目查询、推荐、RAG、任务、反馈、订阅和推送候选。
8. RAG 基础能力，包括 FTS5、本地确定性 embedding、向量检索、混合检索、解释、质量趋势和维护计划。
9. 项目反馈记忆，已经影响推荐排序和项目级 RAG 聚合。
10. 事件订阅与精准推送，支持事件检测、候选构建、确认发送和投递审计。

因此当前最大问题不是“功能太少”，而是“功能之间需要形成稳定的 v1 产品闭环”。

## 三、第一性原理判断

用户真正需要解决的问题不是“看一份 GitHub 周报”，而是：

1. GitHub 项目太多，人工筛选成本高。
2. Star 和 Trending 噪声大，不能直接代表值得跟踪。
3. Agent、RAG、开发工具方向变化快，需要持续观察。
4. 用户需要可解释判断，而不是项目列表。
5. 用户需要在项目变化发生时被提醒，而不是每次重新搜索。

所以 v1 的产品目标应是：

```text
我能打开管理页
-> 输入一个 GitHub 项目研究问题
-> 系统基于历史项目、RAG、反馈和订阅事件回答
-> 我能看到引用证据
-> 我能对项目反馈
-> 我能订阅项目变化
-> 系统每周自动采集、归档、推送
```

这个闭环跑通，就可以认为第一个完整版本成立。更深的项目研究、真实语义 embedding、更复杂 UI、多用户权限和生产级部署都可以后置。

## 四、对抗性审查发现

### 1. Agent 定位仍有功能堆叠风险

当前 README 已经把项目定位成 GitHub 项目研究 Agent，但系统能力仍容易呈现为“周报工具 + 管理接口集合”。如果继续只堆功能，用户很难理解主流程。

修改建议：

1. 后续所有开发都服务三个主流程：项目发现、项目研究问答、项目变化订阅。
2. 新能力必须能说清楚服务哪个主流程。
3. 不再为了“看起来更像平台”提前扩展复杂 UI、多用户或大规模基础设施。

### 2. RAG 当前更像检索层，不是最终竞争力

项目已经有 FTS5、本地 hash embedding、混合检索和解释历史，但这还不是足够强的智能体能力。

主要风险：

1. `local-hash-v1` 不是严格语义 embedding。
2. `/v1/rag/ask` 当前主要是规则版回答。
3. 证据质量主要依赖 README、项目摘要和运行数据。
4. 回答缺少强制证据约束和拒答协议。

修改建议：

1. 先完成证据约束的 RAG Ask，再考虑真实 embedding。
2. 回答必须返回 citations。
3. 证据不足时必须明确拒答或低置信。
4. 增加 RAG golden questions，用固定问题评估回答质量。

### 3. 真实 API 对话有变成普通聊天框的风险

真实 LLM 接入是正确目标，但不能只做“能聊天”。如果只是把问题和上下文交给模型，容易得到泛泛建议，削弱项目差异化。

推荐目标应从：

```text
真实 LLM RAG 对话闭环
```

修正为：

```text
证据约束的 LLM RAG 项目研究问答
```

必须具备：

1. `answer_model` 区分 `llm:kimi` 和 `rule:rag-ask-v1`。
2. 返回 `answer_mode`、`fallback_reason`、`citations`、`evidence`。
3. 无证据时拒答，不强行编造。
4. API 未配置、超时、限流或错误时自动降级。
5. 不把密钥、长原始日志、无关上下文送进模型。

### 4. v1 验收标准不够硬

当前最大的遗漏是没有清楚定义“完整项目”的最小验收标准。没有这个标准，开发会继续扩散。

建议 v1 完整项目只验收：

1. 每周自动采集、归档和链接推送稳定。
2. 管理页可以进行 RAG 问答。
3. 问答可以调用真实 LLM，失败时降级。
4. 回答展示证据、引用、模型状态和降级原因。
5. 项目页可以反馈和订阅项目变化。
6. 管理页可以预览并确认推送候选。
7. 全量测试、安全检查和 GitHub Actions 通过。

不纳入 v1：

1. 复杂多用户权限。
2. 大规模向量数据库。
3. 复杂聊天 UI。
4. 深度项目研究采集。
5. 多模型路由。
6. LangChain 或 Agent 框架大重构。

### 5. `ApiRepository` 有继续膨胀风险

`src/api/repository.py` 已经承载 API 聚合、RAG、任务、反馈、通知等大量逻辑。短期可接受，但继续把 LLM 调用塞进去会增加维护风险。

修改建议：

```text
src/llm/client.py      # 统一模型调用、超时、重试、错误归一
src/llm/prompts.py     # 对话提示词和回答约束
src/rag/answering.py   # RAG Ask 编排、证据约束、降级逻辑
```

`ApiRepository.rag_ask()` 只负责调用编排结果并保持 API 契约稳定。

### 6. 订阅推送需要继续降噪

事件订阅、候选构建和投递审计已经成立。下一步风险是事件太多导致推送噪声。

后续增强：

1. 项目级冷却期。
2. 订阅级最低变化阈值。
3. daily/weekly digest 合并。
4. 反馈“不适合”影响后续候选。
5. 推送前生成“为什么现在提醒我”。

### 7. 集成风险已经高于单点能力风险

当前项目已经连接了周报、API、Pages、RAG、反馈、订阅、推送、Actions 和管理页。下一阶段最容易坏的不是算法，而是集成。

重点注意：

1. API 字段变化导致页面损坏。
2. 任务失败但 Actions 看起来成功。
3. API 未配置时页面无明确错误。
4. 真实发送误触发。
5. 本地可运行但 GitHub Actions 不一致。

修改建议：

1. 所有写入和外发继续保留显式确认。
2. 管理页显示失败原因，而不是静默失败。
3. 每次新增字段同步 API 文档和页面测试。
4. GitHub Actions 中非关键任务可以 `continue-on-error`，但必须记录状态。

## 五、我当前最没有把握的事情

真实 API 接入后，用户体验是否会明显优于规则版 RAG。

不确定点：

1. 当前语料是否足够支撑高质量回答。
2. 召回片段是否覆盖真实用户问题。
3. 模型是否会生成泛泛建议。
4. 管理页对话是否会暴露“看似能聊，实际没有结论”的问题。
5. API 成本和延迟是否可接受。

因此真实 LLM 必须以可观测方式接入：

1. 显示模型名。
2. 显示回答模式。
3. 显示是否降级。
4. 显示证据引用。
5. 显示失败原因。

## 六、用户当前最大遗漏

最大遗漏不是某个功能，而是还没有把 v1 定义成一个可演示、可长期自动运行、可失败降级的产品闭环。

建议先把 v1 定义为：

```text
采集 -> 入库 -> 检索 -> 对话 -> 反馈 -> 订阅 -> 推送 -> 自动运行
```

这个闭环完成后，再逐步强化项目研究深度。

## 七、下一阶段推荐目标

目标名：

```text
证据约束 RAG Ask 接入真实 LLM
```

完成标准：

1. 新增统一 LLM 客户端，优先复用现有 Kimi 配置。
2. `/v1/rag/ask` 支持真实 Kimi API 回答。
3. API 未配置、超时、限流、错误响应时自动回退规则版。
4. 返回 `answer_model`、`answer_mode`、`fallback_reason`。
5. 回答必须带 citations 和 evidence。
6. 管理页显示真实模型状态、引用证据和降级原因。
7. 单元测试覆盖成功、未配置、超时、错误响应、降级。
8. 安全检查确认没有硬编码密钥。
9. README、API 文档、操作日志同步更新。
10. 完成后提交并推送。

## 八、后续发展路线

### v1：完整 Agent 产品闭环

目标：把当前系统打通为可演示、可运行、可降级的产品。

范围：

1. 真实 LLM RAG Ask。
2. 管理页可对话。
3. 项目页反馈和订阅可用。
4. 每周采集和推送保留。
5. 失败时可降级、可查看原因。

### v1.5：订阅推送产品化

目标：让每周推送升级为项目情报订阅模块。

范围：

1. 事件摘要。
2. daily/weekly digest。
3. 推送反馈。
4. 冷却期和降噪。
5. 订阅推荐原因。

### v2：项目研究深度增强

目标：强化项目判断质量。

范围：

1. issue/PR/release 活跃度。
2. 许可证和依赖风险。
3. demo、benchmark、eval 信息。
4. SDK/API 可用性。
5. 同类项目差异分析。

### v3：更强 Agent 工作流

目标：从问答系统升级为任务型研究 Agent。

范围：

1. 项目发现 Agent。
2. 项目研究 Agent。
3. 项目比较 Agent。
4. 风险审查 Agent。
5. 趋势总结 Agent。
6. 订阅推送 Agent。

## 九、近期执行建议

优先顺序：

1. 先做统一 LLM client。
2. 再接 `/v1/rag/ask`。
3. 再加证据约束回答协议。
4. 再改管理页展示。
5. 最后补测试、文档、提交和推送。

暂缓：

1. 深度 GitHub 研究采集。
2. 复杂 UI。
3. 多用户权限。
4. 大规模向量库。
5. 多模型路由。

## 十、最终判断

当前项目方向正确，基础能力已经足够进入 v1 产品闭环阶段。

最大风险不是功能不足，而是功能多但判断不够可信、主流程不够聚焦。下一步必须把真实 LLM 接入做成证据约束的项目研究问答，而不是普通聊天能力。
