# GitHub 项目研究 Agent V2 对抗性审查与路线图

审查日期：2026-07-10  
审查基线：`08b6d67 feat: complete frontend workbench v2`

## 一、结论先行

V2 已经完成了从静态归档页面到 React 项目研究工作台的关键跃迁：用户能够输入自然语言需求，看到流式回答、项目卡、证据、筛选、详情和对比，并且前端交互边界比 V1 清楚得多。

但从第一性原理看，当前系统仍是“带项目证据展示的研究原型”，还不是“可信的项目匹配 Agent”。主要原因不是模型不够强，而是系统尚未证明三个最核心命题：

1. 候选项目集合覆盖了用户真正需要的项目。
2. 排在第一位的项目比其他候选更符合用户约束。
3. 页面显示的 `high/medium` 置信度与真实正确率存在可解释关系。

当前最危险的产品错觉是：系统可以准确检查“回答是否引用了已召回项目”，却不能检查“召回项目是否与问题相关”。引用成立不等于推荐正确，证据数量多也不等于置信度高。

## 二、眼下最没有把握的事情

最没有把握的是：**首选项目是否真的解决了用户的问题。**

这个问题目前无法从代码、测试或运行数据中得到可靠答案。系统没有人工标注的查询与相关项目集合，也没有 Top-1 接受率、Recall@K、MRR、NDCG 或约束满足率。现有检索评估主要比较 FTS5、向量和 hybrid 的命中数量、重叠和零命中情况；它能说明三种模式返回了什么，却不能说明谁返回得正确。

因此，截图中“继续”被推荐到 `CJackHwang/ds2api` 不是偶发 UI 问题，而是当前评估体系无法发现的一类系统性失败。

## 三、最大的遗漏

最大的遗漏是：**没有建立“用户需求 -> 候选召回 -> 排序 -> 回答 -> 用户结果”的可归因质量闭环。**

现有 `project_feedback` 只能记录仓库、profile、评分、标签和备注，不能关联：

- 用户原始问题与解析后的检索问题；
- 本轮候选列表、排名和检索分数；
- 使用的语料版本、embedding 模型和提示词版本；
- 最终回答、引用集合和质量闸门结果；
- 用户为什么接受或拒绝首选项目。

本地审计时 `project_feedback` 仍为 0 条。即使未来有反馈，它也主要改变“这个仓库通常受不受欢迎”，无法学习“这个仓库是否适合这类具体需求”。

## 四、你可能没有意识到的事实

### 1. 当前置信度不是概率，也没有校准

`src/rag/answering.py` 的置信度只看 context 数量：5 条及以上为 `high`，2 条及以上为 `medium`。回答质量分同样由 context、citation 和 evidence 的数量相加得到。

这意味着五条无关或重复证据可以产生 `high`，一条高度相关的直接证据只能得到 `low`。页面上的置信度实际表示“证据块数量”，不表示“推荐正确的概率”。

### 2. `local-hash-v1` 不是语义 embedding

`src/rag/embeddings.py` 将英文 token 和单个汉字哈希到 64 维向量，再计算余弦相似度。它可重复、无需外部依赖，但本质上是带碰撞的词项哈希，不理解同义词、否定、约束关系和用户意图。

当前向量检索只过滤 `score <= 0`；hybrid 再按文本排名 0.55、向量排名 0.45 做名次融合。用户看到“vector/hybrid”时容易理解为语义检索，但实际能力远弱于该名称暗示的水平。

### 3. 质量闸门检查的是格式，不是相关性和蕴含关系

`src/rag/answer_quality.py` 当前检查：回答长度、是否有引用编号、引用编号是否合法、是否提到未知仓库、是否存在 context。

以下回答仍可能通过：

- 使用合法引用，但结论与引用文本无关；
- 推荐一个确实被召回、但不满足用户部署或技术栈要求的项目；
- 使用含 HTML、徽章或导航噪声的证据；
- 把项目 README 中的宣传话术当成已经验证的能力；
- 同一个项目的多条重复证据制造虚假的高置信度。

### 4. 首选项目由召回顺序决定，不是结构化匹配结果

规则降级回答直接选 `contexts[0]`。前端则按 citations/evidence 的出现顺序组装候选，并把第一个候选显示为“最匹配项目”。目前没有独立的 `recommendations[]` 契约来表达约束满足、缺失条件、匹配分和排序理由。

因此，前端的“最匹配”标签比后端实际能证明的结论更强。

### 5. RAG 语料只做空白压缩，没有内容清洗

`src/storage/sqlite_store.py::_clean_text` 只执行空白合并。项目描述、README 摘要、选择原因、风险、项目画像和 Agent 任务记忆会拼接后直接切块。

本地数据库审计快照：

| 指标 | 当前值 |
|---|---:|
| 唯一项目 | 79 |
| 历史 selections | 162 |
| RAG chunks | 261 |
| 含 HTML 标签的 chunks | 18 |
| 含 Markdown 图片的 chunks | 17 |
| 含 HTML 属性的 chunks | 16 |
| 含 badge/Trendshift 文本的 chunks | 21 |
| 重复 chunk 文本组 | 10 |
| 用户项目反馈 | 0 |

这解释了项目卡中出现 `href`、`src`、`width`、徽章链接和 README 残片的问题。

### 6. “全部 GitHub 项目”与“历史归档候选”不是一回事

当前 corpus 来自历史 `selections`。系统擅长在已经采集和入选的项目中排序，不具备对 GitHub 全量项目的覆盖保证。

如果用户问一个归档未覆盖的细分需求，系统仍可能从现有 79 个项目中强行选出第一名。产品文案必须持续强调“当前归档内最匹配”，不能暗示全网最优。

### 7. 本地数据和自动归档存在分支分裂

本地 `data/github_weekly.sqlite` 的最新 selection 日期为 2026-06-03；GitHub 上最近一次周任务在 2026-07-06 成功，产物写入 `weekly-archive` 分支。

这意味着自动任务在运行，但新拉取的 `main` 工作区或长期运行的本地 API 不一定自动获得最新归档。用户可能在不知情的情况下，用五周前的本地数据库测试“当前项目推荐”。

### 8. V2 尚未经过提交触发的 CI

主工作流只由 schedule 和 `workflow_dispatch` 触发；最近一次周任务运行在 V2 提交之前。`secrets-check.yml` 的 push 触发还只监听工作流文件自身。

当前没有针对每次 `main` push 或 pull request 的独立质量门禁，也没有 Playwright 浏览器回归。V2 虽然本地测试通过，但远端还没有自动验证这次提交的前端构建、Python 回归和生成物一致性。

### 9. 对话请求存在隐私和资源边界问题

流式问答使用 GET，请求问题放在 URL 的 `q` 参数中；问题还会连同最终响应保存在 `localStorage`。这适合单机原型，但需求文本可能进入浏览器历史、访问日志或本机其他脚本可读的存储。

只读 RAG/LLM 接口没有明显的速率限制、并发限制或查询长度预算。如果未来监听非本机地址，需要先补齐滥用防护，不能只依赖管理写接口的 Token。

### 10. 外部项目文本仍是未标记的不可信输入

RAG 提示词要求模型只使用 `prompt_context`，但没有明确声明其中的 README/项目文本是不可信数据，也没有把外部文本与系统指令做结构化隔离。当前未发现明显提示注入样本，不代表边界已经安全。

## 五、对抗性场景

| 场景 | 当前可能行为 | 正确行为 |
|---|---|---|
| 用户输入“继续” | 把“继续”作为独立关键词检索 | 继承最近用户意图，或要求澄清 |
| 用户要求“完全离线、不要 API Key” | 召回提到“本地部署”但实际依赖云 API 的项目 | 将离线和无密钥设为硬约束，违反即淘汰 |
| README 含大量徽章/HTML | 噪声进入 chunk、卡片和模型上下文 | 结构化清洗后再索引，原文仅供折叠审计 |
| 召回五条无关证据 | 页面显示 `high` confidence | 显示“证据多但相关性未通过”，拒绝推荐 |
| 仓库被召回但证据不支持结论 | 只要仓库名和引用合法就可能通过 | 对每条关键 claim 做 evidence entailment 检查 |
| 用户描述未被归档覆盖的细分需求 | 从现有候选强行选第一名 | 明确覆盖不足，可选在线扩展召回或拒答 |
| 恶意 README 写入模型指令 | 文本直接进入 prompt_context | 标记为不可信数据，隔离并过滤指令型内容 |
| 本地 API 长期未同步归档分支 | 页面仍显示“可开始研究” | 显示数据日期和过期警告，提供同步动作 |

## 六、整改优先级

## P0：先停止错误确定性

### P0-1 建立人工标注的匹配评估集

创建 `evals/project_match_cases.jsonl`，首批至少覆盖 50 个中文需求，包含：

- 原始问题；
- 必须满足和必须排除的硬约束；
- 可接受项目、明确不相关项目；
- 预期澄清问题；
- 证据最低要求。

输出 Recall@3、Recall@10、MRR@10、Top-1 接受率、硬约束违反率、拒答准确率。没有这个基线，不应继续调 hybrid 权重或提示词。

### P0-2 修正质量和置信度语义

- 将现有 `confidence` 改名或重新解释为 `evidence_coverage`。
- 新增 `match_confidence`，在没有标注数据校准前只能返回 `unknown/low`。
- `answer_quality` 分离为 `citation_validity`、`evidence_relevance`、`claim_support`、`data_freshness`。
- 不再因为 context 数量达到 5 就显示 `high`。

### P0-3 清洗并版本化 RAG 语料

- 使用 Markdown/HTML 解析器移除图片、徽章、HTML 属性、目录、导航和重复模板。
- 项目名称、描述、部署方式、技术栈、许可证、维护状态、限制条件分别存为结构化字段。
- README 原始片段与规则生成的项目画像分开索引，记录 `source_type`。
- 增加 `corpus_version`、`cleaner_version` 和内容哈希；重建后检查噪声率与重复率。
- 对外部内容添加提示注入检测和“不可信数据”标识。

### P0-4 让后端返回结构化推荐，而不是让前端猜第一名

新增非破坏性字段：

```text
recommendations[]
  full_name
  rank
  match_score
  matched_requirements[]
  unmet_requirements[]
  reasons[]
  citation_indexes[]
  evidence_chunk_ids[]
  eligibility: eligible | rejected | unknown
```

前端只能把 `eligibility=eligible` 且通过质量闸门的第一项标为“最匹配项目”。

### P0-5 支持追问解析和澄清

- 最近用户问题和当前候选可以作为“意图上下文”，但历史模型回答仍不能作为事实证据。
- “继续”“那个呢”“更适合 Python 的”先改写成完整检索意图。
- 无法唯一还原时返回 `clarification_required`，给出一个简短追问。
- 对部署、成本、许可证、技术栈等硬约束先解析再检索和重排。

### P0-6 增加提交级 CI 门禁

新增独立验证工作流，触发 `pull_request` 和 `push main`：

- `npm ci`、TypeScript、Vitest、Vite build；
- Python 全量测试、安全检查、`git diff --check`；
- 重建 `docs/app/` 后检查工作树无差异；
- Playwright 桌面/手机：固定输入框、长流式输出、拒答、降级、筛选分页和对比。

## P1：把原型升级为可学习系统

### P1-1 建立查询级反馈和可复现快照

新增 `answer_runs` 与 `answer_feedback`，记录 query、resolved_query、候选排名、语料版本、检索模式、模型、提示词版本、引用和最终状态。反馈必须能回答“哪个问题下，哪个推荐为什么不合适”。

### P1-2 替换或诚实命名本地向量检索

两种可接受路径：

1. 接入真实多语言 embedding，并记录模型、维度和索引版本；
2. 保留 `local-hash-v1` 作为 lexical-hash baseline，在 UI 和 API 中明确命名，不宣称语义向量能力。

先用评估集证明提升，再决定是否引入外部向量数据库。

### P1-3 建立数据新鲜度契约

- API 返回 `archive_updated_at`、`corpus_built_at`、`embedding_built_at`。
- 超过 8 天未更新时，Agent 页显示过期警告并降低推荐资格。
- 提供从 `weekly-archive` 同步本地数据的标准 PowerShell 命令或只读管理动作。
- 健康检查不能只判断 API 可访问，还要判断数据是否新鲜。

### P1-4 增加隐私和滥用保护

- 限制问题长度、并发、单 IP/单进程速率和上游模型预算。
- 生产化时改为 POST 流式请求，或先 POST 创建短期 request ID 再订阅 SSE。
- 提供“无痕会话”和本地历史保留期限；默认不在 URL 中暴露完整需求。
- 日志只记录请求 ID、耗时和分类，不记录完整问题。

## P2：扩大项目覆盖和工程可维护性

### P2-1 按需扩展候选宇宙

当本地归档覆盖不足时，可选调用 GitHub Search 扩展候选，再进入同一清洗、风险、证据和评估流程。在线召回必须显式标记，不能与历史归档事实混在一起。

### P2-2 建立长期质量看板

按语料版本、检索版本和模型版本跟踪：Top-1 接受率、Recall@K、澄清率、拒答率、硬约束违反率、降级率、P50/P95 延迟和成本。

### P2-3 拆分超大模块

`src/api/repository.py` 已同时承担项目查询、RAG、评估、反馈、任务、通知和维护逻辑。按领域拆分 repository/service，但保持 API 契约不变，避免后续每次改检索都承担跨模块回归风险。

## 七、建议实施顺序

```text
评估集与基线
-> 修正 confidence/quality 文案和契约
-> 语料清洗与重建
-> 结构化 recommendations
-> 追问解析与澄清
-> 真实 embedding 对照实验
-> 查询级反馈闭环
-> 数据新鲜度与隐私保护
-> 按需扩大候选覆盖
```

不要先做多 Agent 编排、复杂长期记忆或更强模型。当前瓶颈是检索与评估的可证明性，不是 Agent 数量。

## 八、阶段验收标准

P0 完成至少满足：

1. 评估集不少于 50 条，并可重复运行。
2. HTML、Markdown 图片和徽章噪声 chunk 数降为 0。
3. “继续”在有上下文时正确解析，无上下文时触发澄清。
4. 硬约束违反的项目不能成为首选。
5. 前端首选项目只来自后端结构化 `recommendations`。
6. 不再用证据数量直接生成 `high` 置信度。
7. push/PR 自动运行前后端测试、构建和浏览器关键流程。
8. 页面显示归档和索引更新时间，过期数据不可伪装为当前结论。

首轮指标阈值应在建立基线后确定。可以先以 Recall@10 >= 0.85、MRR@10 >= 0.60、硬约束违反率 <= 5% 作为实验目标，但不能在没有人工标注数据时宣称已经达到。

## 九、最终判断

V2 的前端工作值得保留，它已经提供了可承载真实研究能力的产品外壳。下一阶段不应继续把主要精力放在视觉细节，而应让“最匹配”成为一个可测量、可复现、可被用户纠正的后端决策结果。

本项目当前最需要的不是更多功能，而是一个不会把“有引用”误认为“结论正确”的质量体系。

## 十、执行注意事项

1. 先建立评估基线，再调检索、提示词或模型。否则每一次“效果变好”的判断都只是主观体验。
2. 不要为了支持连续对话而把历史模型回答塞回 RAG 证据。可传递最近用户意图和当前候选，但事实只能来自本轮可审计语料。
3. 所有语料清洗、embedding、排序和提示词改动必须携带版本号，并在 `answer_runs` 或评估结果中留下快照；否则出现回归时无法定位原因。
4. 语料重建是破坏性派生操作。先在临时 SQLite 或备份上统计 HTML 噪声、重复率、chunk 数和检索指标，再替换正式索引。
5. 把“当前归档内最匹配”写进产品文案、API 和状态标签。候选覆盖不足时应澄清或拒答，不能把有限归档包装成全网搜索。
6. 接入真实 embedding 前，先用同一评估集和固定语料做对照实验；若指标没有提升，不要增加外部服务、成本和运维复杂度。
7. 查询级反馈应以可选、最小化数据为原则。记录评分和失败原因足够，不要默认持久化用户完整业务需求或敏感上下文。
8. 上线到非本机地址前，必须补齐认证边界、速率限制、并发预算、日志脱敏和请求长度限制；管理 Token 只保护写接口并不能保护模型成本。
9. 每次发布前同时验证两条数据路径：本地 API 使用的 SQLite 数据，以及 `weekly-archive` 发布的归档数据。两者日期不一致时，应在页面明确展示，而不是静默继续推荐。
10. P0 完成前，不要把“高置信度”“智能推荐”作为对外承诺。更诚实的定位是“证据约束的项目研究辅助工具”。
