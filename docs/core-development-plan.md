# 核心功能优先建设路线

本文根据 `deep-research-report (4).md` 重新整理 GitHub Weekly Agent 的后续建设方向。新的原则是：先把项目建设成稳定的“后端 + Agent 核心”，再逐步增强前端、展示和末端体验。

## 一、定位调整

当前项目已经不只是一个 GitHub 抓取脚本，而是一个开源项目情报系统雏形。它已经具备：

1. GitHub Trending 与 GitHub Search 采集。
2. 每周自动运行与手动触发。
3. Kimi 生成中文周报与规则版降级。
4. JSON 归档、SQLite 派生索引和 GitHub Pages 展示。
5. FastAPI 只读接口。
6. Telegram、飞书、企业微信等推送入口。
7. 个性化 profile、质量信号、基础风险提示和历史查询。

后续建设重点应从“继续修页面细节”转向“补齐后端服务化、数据层、Agent 分析和任务调度能力”。

## 二、建设优先级

### P0：主链路稳定

目标：每周热点追踪、周报生成、归档和推送稳定运行。

重点任务：

1. 保持 GitHub Trending 为第一热点信号。
2. 保持新增 Star、近期活跃度、质量信号和个性化 profile 作为辅助信号。
3. 保持 Kimi 失败后的规则版周报降级。
4. 保持 GitHub Actions 自动运行与 `weekly-archive` 独立归档分支。
5. 保持所有运行摘要可追踪、可复盘。

暂不优先：

1. 复杂视觉效果。
2. 重型前端框架。
3. 多租户权限系统。
4. 微服务拆分。

### P1：后端服务化核心

目标：让项目从“定时任务”升级为“可查询、可触发、可回放”的后端系统。

优先接口：

1. `GET /v1/projects`：项目检索、语言过滤、profile 过滤、时间过滤。
2. `GET /v1/projects/{owner}/{repo}`：项目详情、历史趋势、相似项目。
3. `GET /v1/runs`：运行记录和失败原因。
4. `POST /v1/runs/trigger`：手动触发一次采集和周报任务。
5. `GET /v1/jobs/{job_id}`：查询长任务状态。
6. `GET /v1/reports/latest`：读取最新结构化周报。

实现原则：

1. 先复用当前 FastAPI，只做兼容扩展。
2. 保留现有 `/api/*` 只读接口，新增 `/v1/*` 作为正式服务接口。
3. 长任务返回 `job_id`，不把采集和 LLM 生成阻塞在单次 HTTP 请求里。

### P2：数据层升级

目标：从“公开归档文件”升级为“可持续查询和分析的数据底座”。

短期：

1. JSON 继续作为公开归档和审计源。
2. SQLite 继续作为本地派生索引和轻量查询层。
3. 增强迁移校验，确保 SQLite 可由 JSON 重建。

中期：

1. 引入 PostgreSQL 作为正式业务主库。
2. 使用 JSONB 保存项目原始元数据和运行摘要扩展字段。
3. 保留 JSON/SQLite 导出，避免数据被锁死在数据库中。

后期：

1. 使用 pgvector 承接项目摘要、README 摘要、周报片段和反馈的向量检索。
2. 数据量明显增长后，再评估 Qdrant 或 Elasticsearch。

### P3：Agent 与 RAG 核心

目标：让系统不仅能生成周报，还能围绕历史项目做可解释分析。

核心能力：

1. Planner：判断用户问题需要结构化查询、RAG 检索还是工具调用。
2. Tool Registry：统一封装项目查询、历史趋势、报告读取、相似项目、推送通道。
3. Retriever：先 SQL 过滤，再全文检索，最后向量检索。
4. Evaluator：检查回答是否包含证据、项目链接、时间范围和结构化字段。
5. Memory：保存 profile、用户反馈和长期关注方向。

优先实现的 Agent 场景：

1. “最近四周 Python Agent 项目趋势如何？”
2. “给我推荐适合学习 Java 后端的热门项目。”
3. “对比三个类似项目，说明哪个更值得深入研究。”
4. “根据我的 profile，解释本周为什么推荐这些项目。”

暂不优先：

1. 完整通用聊天机器人。
2. 大而全的多 Agent 框架。
3. 复杂 MCP 生态接入。

### P4：异步任务与调度

目标：支撑按需触发、后台运行、状态查询和失败重试。

短期：

1. 用本地任务表或 SQLite job 表实现最小任务状态。
2. 支持 `queued`、`running`、`succeeded`、`failed`、`cancelled` 状态。
3. 支持任务日志摘要和错误原因归档。

中期：

1. 轻量引入 APScheduler 做本地调度。
2. 引入 Redis 作为任务锁、限流和缓存。
3. 任务复杂后再评估 Celery。

### P5：前端与产品体验

目标：前端服务核心功能，而不是先做复杂视觉。

优先页面：

1. 项目检索页：语言、profile、时间、来源、质量分、风险提示过滤。
2. 项目详情页：历史趋势、相似项目、推荐原因、风险提示。
3. 运行状态页：采集成功率、Kimi 状态、推送状态、错误分类。
4. Agent 分析页：输入问题，展示证据、引用项目和结论。

暂不优先：

1. 营销式首页。
2. 大型 SPA 重构。
3. 动效和视觉装饰。

## 三、推荐技术栈顺序

按引入顺序：

1. FastAPI：继续作为后端入口。
2. SQLite：继续作为本地派生索引。
3. PostgreSQL：在 `/v1/*` 和任务状态稳定后引入。
4. Redis：用于任务状态、缓存、限流和锁。
5. pgvector：用于项目语义检索和 RAG。
6. APScheduler：用于轻量定时和按需任务。
7. Celery：只有当任务队列明显复杂后再引入。
8. 前端框架：只有当静态页面无法支撑交互后再引入。

## 四、下一阶段执行清单

最近优先做：

1. 设计 `/v1/*` API 契约文档。
2. 增加最小 job 模型和任务状态查询。
3. 把当前 `main.py` 主流程封装成可复用 use case。
4. 增加手动触发周报任务的后端入口。
5. 设计 PostgreSQL schema 草案，但暂不立即迁移生产数据。
6. 设计 Agent 分析的输入输出 schema。
7. 增加项目历史检索的证据字段，为 RAG 做准备。

暂缓：

1. 继续美化 GitHub Pages。
2. 继续扩展推送渠道。
3. 继续细化安全评分。
4. 过早引入前端工程化。
5. 过早拆分微服务。

## 五、验收标准

下一阶段完成后，项目应具备：

1. 可以通过 API 手动触发一次完整采集任务。
2. 可以查询任务状态和失败原因。
3. 可以按 profile 查询历史项目。
4. 可以解释某个推荐项目的证据来源。
5. 可以为 Agent/RAG 提供结构化项目上下文。
6. 周报主链路仍然稳定，不因新能力影响每周自动运行。
