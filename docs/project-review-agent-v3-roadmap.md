## 一、审查口径

本报告从项目的最终目标倒推：用户需要的不是“能返回仓库卡片的聊天界面”，而是一个能在**正确的问题类型、正确的候选范围、足够新鲜的证据和可解释约束**下，帮助其发现、理解、比较和选择 GitHub 项目的研究 Agent。

报告使用以下标记，避免把推测写成事实：

- **【事实】**：可以由当前代码、测试、固定评估、SQLite、Git 或远端只读状态直接复核。
- **【推断】**：由事实推导出的高概率风险，但尚未被真实流量或独立样本证明。
- **【待验证】**：当前证据不足，必须通过独立实验、真实样本或人工审计回答。

### 1. 证据范围

本次检查了：

- 当前代码、Prompt、FastAPI 契约、SQLite 契约和 React 行为；
- V1/V2 路线图、README、架构、API、操作日志和最近提交；
- 52 条项目匹配、52 条结构化推荐、60 条追问、100 条约束解析和 60 条约束证据评估；
- Python、Vitest、mock Playwright、真实 FastAPI + 临时 SQLite Playwright、安全检查和前端构建；
- 用户手测中的“对比这两个”“学习 Agent 开发”问题，以及额外的未见同义表达；
- 本地 SQLite、语料版本、embedding 来源、远端 `main`、`weekly-archive` 和 Git 对象完整性。

本次**没有**调用真实 Kimi、GitHub 采集、Telegram、飞书、企业微信或其他真实外发功能，也没有把模型知识当作项目证据。

### 2. 已经证明的能力

【事实】V2 以来的工作不是无效的。当前项目已经证明：

- Ask GET/POST 和 SSE 契约有较完整的单元及真实链路回归；
- 无上下文短追问可以保守澄清，显式序号可以限制候选范围；
- 已覆盖的确定性硬约束反例不会把明确冲突项目判为合格；
- 管理口令不再从 URL 或浏览器持久化存储中使用；
- mock 与真实后端 Playwright 能隔离界面状态错误和 FastAPI/SQLite 集成错误；
- `push main` 和 pull request 都有提交级自动检查；
- `confidence` 已被降级为兼容字段，界面不再把证据数量包装成匹配概率。

这些能力证明了工程回归基础和部分安全边界，**不能自动推出真实用户任务已被正确解决**。

## 二、结论先行

当前系统已经从“带 RAG 的展示原型”发展为“有确定性约束、真实后端回归和可审计候选结构的工程原型”，但还不能称为经过校准的项目研究 Agent。

根本原因不是 Kimi 上下文不够长，也不是缺少 Agent 框架，而是以下四个命题尚未同时成立：

1. 系统先判断用户是在搜索、解释、比较、学习还是补充条件，再选择是否检索。
2. 未校验的模型内容不会先于质量闸门到达用户。
3. 当前检索使用的数据确实新鲜、完整、清洗版本正确，并且公开归档没有混入私有运行态。
4. 固定回归集之外的真实表达仍能保持正确路由、约束、候选范围和最终答案。

因此，下一阶段不应直接进入查询级反馈，也不应先迁移 Responses API、Agents SDK 或调 hybrid 权重。正确顺序是：

```text
公共/私有归档边界与 Git 完整性止血
→ 安全流式输出
→ 顶层意图、回答类型与证据策略
→ 纯读 API、模型白名单和数据新鲜度
→ 独立 blind 评估与查询级决策闭环
→ 真实 embedding 对照和模块拆分
```

## 三、眼下最没有把握的事情

眼下最没有把握的是：**由实现者共同维护的固定评估集，是否代表真实用户的自然语言分布。**

### 当前证据

- 【事实】52 条项目匹配样本只对应 8 个合成仓库；48 条可检索样本本质上是每个主题的多次模板改写，另有 4 条通过不存在的筛选值产生零命中。
- 【事实】语料只有 8 个项目，评估却计算 Recall@10，因此 Recall@10 在该集合上几乎没有真实排序压力。
- 【事实】所谓 locked/adversarial 集与实现修改由同一开发过程共同维护，适合回归，不是独立 holdout。
- 【事实】只有约束解析提供可选 `--blind-dataset` 入口，当前没有项目匹配、追问、证据分类和最终回答的独立 blind 结果。
- 【事实】真实 E2E 会主动清空 Kimi 环境变量，所以它证明降级链路稳定，不证明真实模型面对提示提取、语料注入和混合意图时的行为。

### 对抗性反例

- “对比一下这两个呢，他俩的优点和缺点分别是？”被判为 `new_search/archive`，上一轮两个候选不再约束检索。
- “我现在想学习 Agent 开发，你告诉我应该怎样学习”被判为 `new_search/archive`，系统返回项目推荐而不是学习路径或澄清。
- “不想接第三方大模型服务”中的“第三方”会被序号正则当成“第三个候选”。
- “Docker 可有可无”会被硬化为必须包含 Docker。
- 额外 8 条探索性同义表达中，`local-hash-v1` Top-3 只命中 4 条、Top-1 只命中 1 条。该小样本不是正式 benchmark，但足以反证“固定集满分等于自然语言已解决”。

### 最小验证动作

在修复这些表达前，由不参与本阶段规则调参的人冻结 24–40 条原始输入，至少覆盖搜索、解释、比较、学习指导、真实 no-match、混合意图、复数指代、可选条件和同义否定；保留首次运行结果，修复时不得同步改标签。

## 四、当前最大的遗漏

### 最大的产品机制遗漏：没有检索前的顶层决策层

当前 `follow_up_router` 解决的是“本轮和上一轮是什么关系”，不是“用户希望得到什么类型的答案”。

【事实】路由只有 `new_search | resume | refine | clarify`，没有 `search | explain | compare | learning_plan | unsupported` 等回答类型。所有较长且未命中特例的输入默认进入 `new_search/archive`。规则降级回答也只有通用候选推荐模板。

系统目前把下列概念混在一起：

```text
conversation_relation   本轮与上一轮的关系
answer_type             用户要搜索、解释、比较还是学习指导
retrieval_scope         全归档、上一轮候选、单个项目或无需检索
retrieval_query         实际用于召回的文本
resolved_user_request   保留当前动作的完整用户意图
evidence_policy         该回答类型需要什么证据，哪些主张禁止输出
```

这解释了为什么“给两个项目做对比”会重新搜索：系统只判断这是不是一个足够长的新问题，没有先判断它是一个指向现有候选的比较动作。

### 比产品遗漏更紧急的系统边界遗漏：没有公共归档投影

【事实】`scripts/publish_archive_branch.py` 会复制并提交整个 `docs/`、`reports/` 和 `data/`；审查时公开仓库的远端 `weekly-archive` 确实跟踪 `data/github_weekly.sqlite`，文件约 24 MB。当前 schema 的同一 SQLite 可包含 RAG 查询、回答、`prompt_context`、反馈备注、订阅、任务、通知和投递审计。

【待验证】本次没有下载并逐表检查远端数据库内容，因此不能宣称它已经包含真实敏感用户数据。

【推断】只要运行态数据库将来写入真实查询、反馈或订阅，现有发布器就可能把它们带入公开归档。这是不可通过 `.gitignore` 解决的发布边界问题，也比一般产品误推荐更难撤回。

## 五、你可能没有意识到的事实

### 1. 最终质量闸门不能保护已经展示的草稿

【事实】`src/rag/answering.py:174-181` 先逐段 `yield delta`，聚合完整回答后才调用 `validate_rag_answer()`；前端收到 delta 后立即显示。质量失败只能用后续 `final` 替换草稿，不能撤回用户已经看到、截图或记录的内容。

【事实】模型回答还会在校验前由 `_ensure_citation_marker()` 自动补一个合法引用。现有质量闸门只检查长度、引用编号、未知仓库和 contexts 是否存在，且明确返回：

- `evidence_relevance = not_evaluated`
- `claim_support = not_evaluated`
- `data_freshness = unknown`

【推断】提示复述、无关结论或不受证据支持的文字可能先被展示，甚至在自动补引用后通过最终格式校验。真实 Kimi 是否会在特定攻击下泄露内部指令仍是【待验证】，但当前链路缺少防护是【事实】。

### 2. 已实现新版清洗器，不等于正在使用新版语料

【事实】当前本地 SQLite 的 162 条 corpus 和 261 条 chunk 全部仍为 `legacy-v0`。只读审计发现 122 处 HTML 标签、46 处 Markdown 图片、71 处 badge URL 和 74 处 HTML 属性残留；当前代码期望 `project-corpus-v2` 与 `stdlib-markdown-html-v1`。

【事实】本地业务数据最大 `run_date` 为 2026-06-03，embedding 虽在 2026-07-08 重建，来源仍是 6 月 3 日。远端 `weekly-archive` 审查时已更新到 2026-07-13。

【推断】“embedding 已存在”“数据库文件今天有修改”都不能证明底层来源新鲜。当前 Ask 如果使用本机数据库，仍可能在旧且含噪语料上工作。

### 3. 若干 1.0 指标是回归证据，不是校准证据

【事实】项目匹配测试主要检查指标字段存在且位于 `[0,1]`，即使 Recall/MRR 降为 0 也可能通过；推荐评估只对硬约束违反率设置强断言。

【事实】追问评估不执行真实检索，“原始追问检索违反率”通过比较字符串计算；比较请求的样本甚至允许 `resolved_query` 被上一轮目标覆盖。

因此，“固定规则集通过”应继续使用，但不得表述为“能力模型已经校准”或“自然语言追问已经解决”。

### 4. 无鉴权 GET 并非全部纯读

【事实】`GET /v1/rag/explain` 和 GET Ask 会持久化 `rag_explanations`；vector/hybrid GET 可通过 `auto_build=true` 写 embedding。`model` 没有严格白名单，不同模型名最终仍调用同一个本地哈希算法。

【推断】如果服务暴露到 loopback 之外，攻击者可用大量模型标签制造重复 embedding 和 CPU/磁盘消耗；即使只监听本机，GET 具有隐藏副作用也会破坏缓存、重试和只读安全假设。

### 5. `local-hash-v1` 的名称可能让人高估语义能力

【事实】它是 64 维 token feature hashing，中文按单字进入 SHA-1 分桶；不学习同义词、词序、否定或上下文。任意其他 `model` 标签目前也不会切换到真实 embedding 实现。

【推断】hybrid 与 local-hash 在固定评估中结果完全相同，说明当前样本没有证明两者互补，也没有证据支持现在调整 hybrid 权重。

### 6. “浏览器本地保存”仍然保存得比最小上下文更多

【事实】React 会把有限数量会话的完整 final `RagAnswer` 保存在 localStorage，其中可包含 route、contexts、citations、evidence、recommendations 和 `prompt_context`。POST 追问只回传最小用户上下文、不把历史 assistant answer 当证据，这一边界仍然成立。

【推断】本地保存不等于数据最小化；同源脚本、XSS、共享电脑或浏览器备份仍可能读取完整历史和证据。

### 7. CI 已建立，但它发生在直接推送之后

【事实】提交级 CI 已覆盖 main push 和 PR，最近远端运行通过；项目明确选择暂不启用分支保护。因此“没有 CI”已不是缺口，“CI 不能阻止一个失败提交先进入 main”是已接受风险。

真正仍未解决的是：检索核心指标没有冻结阈值门禁，以及真实 Kimi 不在任何确定性 E2E 中。

### 8. 当前 Git 可达历史可读，但对象库并非完整健康

【事实】`git fsck --connectivity-only --no-dangling --no-reflogs` 通过，HEAD 和当前可达树可读；但 full fsck 发现损坏的不可达 loose object `0d5add93193596c7597490811fc0cd23952335aa`，并导致一次 fetch 解包失败。

【推断】它更像不可达或重复对象损坏，不足以证明 main 已损坏；但在备份和确认可恢复副本前，不应直接删除对象，也不能宣称仓库 full-fsck 健康。

## 六、主要对抗性失败场景

| 场景 | 当前失效路径 | 可能后果 | 当前测试为何没拦住 |
| --- | --- | --- | --- |
| “对比这两个/他俩” | 复数指代未识别 → `new_search/archive` | 新项目替换原候选 | 追问集无复数指代和回答类型 |
| “比较第一个和第二个” | 范围选对 → 当前请求被旧目标覆盖 | 返回两张卡但没有真正比较 | 测试只断言候选范围 |
| “我想学习 Agent 开发” | 长文本默认新搜索 | 回答项目推荐而非学习路径 | 没有 learning intent |
| “不接第三方模型服务” | “第三方”命中序号正则 | 错选第三个候选 | 固定集没有该自然表达 |
| 模型输出内部规则或无关草稿 | 原始 delta 先显示 → 最终再校验 | 无法撤回的内容暴露 | real E2E 不调用 Kimi，单测接受先草稿后替换 |
| 无引用模型答案 | 自动补 `[1]` → 格式校验通过 | 引用外观掩盖无支持结论 | 闸门不验证主张蕴含关系 |
| `auto_build=true&model=random-*` | 公共 GET → 每个标签生成整套 hash embedding | 写放大、伪 provenance | 没有模型白名单和只读不变性测试 |
| 运行态 SQLite 进入归档 | 整包复制 `data/` → push 公共分支 | 查询、反馈或订阅可能进入 Git 历史 | 测试只验证允许目录，不验证文件白名单 |
| 本地索引“已就绪” | 只看数量/构建时间 → 忽略来源日期和语料版本 | 在过期含噪证据上作答 | diagnostics 缺少来源水位和完整度 |

## 七、整改优先级

### P0：先停止错误确定性和难以撤回的风险

#### P0-0：公共归档改为显式白名单投影

- **修改**：停止向 `weekly-archive` 发布运行态 SQLite；只发布明确可公开的报告、静态页面、去标识化 JSON 和必要的来源快照。为归档发布器加入文件 allowlist、敏感 canary 测试和 staged tree 审计。
- **影响**：极高；**紧急性**：立即；**可逆性**：新发布可逆，既有 Git 历史部分不可逆。
- **依赖**：先决定私有运行态 SQLite 的持久化位置；审计远端历史中是否存在真实查询、反馈、订阅或令牌痕迹。
- **验收**：归档树不再出现 `*.sqlite*`；植入临时敏感标记后，发布预览和 staged tree 均找不到它；若发现真实敏感内容，另行制定历史清理与凭据轮换方案。

#### P0-1：安全流式输出和真实引用校验

- **修改**：先完整缓冲 provider 输出，执行策略检查、引用检查和质量校验，通过后再按现有 `meta → delta* → final` 契约切片发送；不转发原始 provider delta。移除“先自动补引用、再判通过”的路径。
- **影响**：极高；**紧急性**：立即；**可逆性**：可逆；**代价**：首个可见 token 延迟上升。
- **依赖**：定义禁止输出、主张引用和超长回答策略。
- **验收**：恶意假模型返回的提示复述、无引用结论和跨 delta 禁用文本不得出现在任何 SSE 帧、DOM、控制台或 final；普通 POST 与 SSE final 仍等值。

#### P0-2：顶层意图、回答类型和证据策略

- **修改**：在检索前新增结构化决策对象，分离 `conversation_relation`、`answer_type`、`retrieval_scope`、`retrieval_query`、`resolved_user_request` 和 `evidence_policy`；至少支持 search、explain、compare、learning_plan、clarify、unsupported。
- **影响**：极高；**紧急性**：立即；**可逆性**：可通过兼容字段渐进接入。
- **依赖**：先冻结本次手测及独立 blind 样本；不新增服务端会话，不提交历史 assistant 文本。
- **验收**：已知反例 100% 正确；“这两个/这俩/他俩”只绑定明确的两个候选；“第三方/第一版/第二阶段”不得触发候选序号；比较回答必须包含双方同维度证据，学习指导不得静默进入全归档搜索。

#### P0-3：恢复 GET 只读语义并限制模型预算

- **修改**：embedding 构建和解释持久化移到显式管理 POST 或 planned job；GET 默认不写库、不触发 Kimi 费用；`model` 使用实现能力白名单，统一限制查询长度、速率、并发和构建预算。
- **影响**：高；**紧急性**：立即；**可逆性**：可逆。
- **依赖**：保留现有 Ask 响应兼容；明确哪些分析需要审计写入。
- **验收**：所有 GET 前后 SQLite 文件哈希及关键表计数不变；100 个随机模型名不能新增 embedding；未授权请求不能触发模型或构建任务。

#### P0-4：同步来源并重建当前活跃语料

- **修改**：先把最新归档导入临时库，再执行 v2 清洗、分块、噪声审计和 `local-hash-v1` 重建；验证后以可回滚方式替换本机派生索引。Ask 发现来源过期或语料版本不兼容时，不得显示“已确认首选”。
- **影响**：高；**紧急性**：立即；**可逆性**：临时库验证和备份后可逆。
- **依赖**：先修复/绕开本地 Git 损坏对象并确认远端来源；不在只读请求中自动重建。
- **验收**：corpus/cleaner 版本全部为当前版本，HTML、图片、badge、属性残留和完全重复 chunk 均为 0；来源水位、归档 commit、导入时间和构建时间可查询；四套回归指标不下降。

#### P0-5：修复 Git 对象库并冻结关键指标

- **修改**：备份 `.git` 和损坏对象，确认远端或 pack 中的可恢复副本后再修复，最终运行 full fsck；同时将 Recall@3、MRR@10、Top-1、错误接受率和硬约束违反率写成有基线容差的 CI 断言。
- **影响**：中高；**紧急性**：现在；**可逆性**：Git 对象修复前必须备份。
- **验收**：fetch 正常、full fsck 通过；核心指标降到阈值以下时 CI 必须失败，而不是只验证数值位于 `[0,1]`。

### P1：建立独立证据和可学习闭环

1. **独立 blind 评估**：从原始用户输入开始，覆盖 intent、answer type、约束、候选范围、Top-k、eligibility 和最终答案支持度；允许无正确仓库和多个正确仓库，报告 confusion matrix、macro-F1、错误接受率和范围扩张率。
2. **查询级决策快照与反馈**：记录 request ID、结构化意图、候选顺序、route、约束判定、语料/embedding/prompt 版本、来源水位和最终模式；反馈与该快照关联。默认不保存完整 Prompt 或历史 assistant 文本，并先完成公共/私有存储拆分。
3. **真实质量维度**：实现 `evidence_relevance`、`claim_support` 和 `data_freshness`，为关键主张绑定具体 evidence chunk，而不是只检查回答末尾存在 `[1]`。
4. **数据新鲜度与完整度契约**：同时公开来源最大日期、归档 commit、采集错误/partial 状态、导入时间、清洗版本、embedding 构建时间和内容哈希；“success”不得掩盖部分采集。
5. **真实多语言 embedding shadow 对照**：在冻结 blind 集和更大真实语料上与 local hash 并行比较召回、延迟、成本和硬约束结果；达标前不替换 fallback，也不调 hybrid 权重。
6. **模型安全集成回归**：使用本地假 Kimi HTTP/SSE 服务覆盖超时、非法 JSON、提示提取、语料注入、跨 delta 恶意内容和引用欺骗；无需业务密钥或外网。
7. **浏览器数据最小化**：为本地会话提供无痕模式、TTL 和保存字段白名单，默认不持久化 `prompt_context` 与完整 contexts。

### P2：规模化与可维护性

1. 保持 `ApiRepository` 外观契约，按项目查询、RAG、任务、通知、反馈和开发上下文逐域抽取 repository/service。当前 `src/api/repository.py` 已有 8,219 行，职责和回归半径过大。
2. 扩大候选覆盖与历史采样，加入真实 no-match、多正确答案、长尾语言和不同项目生命周期，不再用 8 个合成项目代表真实搜索空间。
3. 建立检索和回答的运行指标：候选覆盖、stale rate、fallback rate、clarification precision/recall、false eligible、人工接受/拒绝原因和每请求成本。
4. 团队或自动部署边界扩大后，再重新评估 required checks 和分支保护；当前暂不启用是已明确接受的风险，不伪装成遗漏。

## 八、建议实施顺序

### 阶段 A：先保护不可逆边界

1. 暂停发布原始 SQLite，审计 `weekly-archive` 当前内容和历史。
2. 备份并修复本地 Git 对象库，恢复可靠 fetch。
3. 把公开归档改成 allowlist 投影并补测试。

### 阶段 B：让用户只看到通过校验的内容

1. 建立本地恶意 provider fixture。
2. 改为完整缓冲、原始回答校验、通过后再发送 delta。
3. 在不改变 SSE 事件名和顺序的前提下完成浏览器回归。

### 阶段 C：修正用户手测暴露的产品根因

1. 冻结独立样本和本次反例。
2. 引入顶层 intent/answer type/evidence policy。
3. 实现复数指代、结构化比较和学习指导/澄清边界。
4. 验证当前问题不会被上一轮目标覆盖。

### 阶段 D：恢复数据可信度

1. 在临时库同步最新来源。
2. 重建清洗语料和 embedding。
3. 将新鲜度、完整度和版本不兼容显式进入诊断及回答状态。
4. 恢复 GET 纯读和模型白名单。

### 阶段 E：再建立查询反馈和模型对照

先做独立 blind 评估与最小决策快照，再接查询级反馈；最后用相同冻结集比较真实多语言 embedding。Responses API、Agents SDK 和服务端聊天会话仍不在这条关键路径上。

## 九、阶段验收标准

只有同时满足下列证据，才可把系统从“工程原型”升级描述为“可信项目研究 Agent 的受控 beta”：

- 公开归档不含运行态 SQLite，既有历史完成敏感内容审计；
- 任何未通过校验的 provider 文本都不会出现在 SSE、DOM、日志或持久化历史中；
- search、explain、compare、learning_plan、clarify 在独立 blind 集上有分类指标，已知反例全部通过；
- 复数指代和序号比较不会扩大候选范围，比较结论对双方使用对称证据；
- 所有公共 GET 都通过数据库不变性测试，模型与构建操作有鉴权和预算；
- 活跃语料使用当前清洗版本，噪声审计通过，来源水位和部分采集状态可见；
- 项目匹配、推荐、追问和约束评估有冻结基线及 CI 阈值；
- `evidence_relevance`、`claim_support`、`data_freshness` 不再是占位值；
- 查询级反馈能回放当时的结构化决策，但不保存完整历史 assistant 文本作为证据；
- Git fetch 与 full fsck 均健康。

## 十、最终判断与执行注意事项

【事实】当前系统的契约测试、确定性约束、管理口令边界和真实后端 E2E 已明显强于 V2 审查时的状态；这些成果应保留，不需要推倒重来。

【推断】眼下用户手测失败的根因不是“需要把完整聊天历史交给 Kimi”，而是后端在调用模型前就选错了回答类型和检索范围。传更多历史最多掩盖部分错误，还会增加提示注入、隐私和不可复现风险。

【待验证】真实 Kimi、真实用户分布、真实多语言 embedding 收益和公开归档数据库的实际敏感内容都尚无充分证据。后续报告必须继续诚实标注这些未知，不得用固定集满分替代。

执行时继续遵守以下边界：

- 不改变既有 Ask/SSE 契约时优先使用非破坏性新增字段；
- 不建立服务端聊天会话，不把历史 assistant 回答当作事实证据；
- 模型增强不能决定硬约束通过；
- 所有真实外发继续保留显式确认；
- 先基线、再改实现；先临时库、再替换本机数据；
- 不启用分支保护是当前已接受风险，但不能因此降低 CI 指标门禁；
- 修复 Git 或归档历史前必须备份，不能直接删除损坏对象或强改远端历史。
